## 1. RULES THE BUILD MUST HONOR

- **Cascade law:** Pillars → Value Goals → Outcome Goals → Projects → Action Items. Every action traces up to the Pillar; the **sum of a goal's projects** advances it.
- **3–5 active-project cap, strict.** Exactly **3 Active** projects (A/B/C). **2 slots held open by design** — splitting focus is the #1 stall. Slots 4–5 stay empty until Project A clears.
- **Minimize active Outcome Goals.** Exactly **2 Active** (OG-1, OG-2). The rest are Next-Up / Future so they can't pull focus early.
- **Every active project has ≥1 active action item** (else it's "stalled").
- **DO-dates, not due-dates.** When you'll *do* it, as a relative anchor.
- **Dates are SET** off a **Go-Live target of Sun 2026-06-07** (the owner's 5-day graduation target; Day 0 = Tue 2026-06-02). Every date below is concrete. The relative anchors (Day N / "from Go-Live") are retained in parentheses so that **if graduation slips, shift every Go-Live-anchored date by the same delta** — except the **domain-warmup clock, which does NOT move** (it's an external ~21-day clock that runs from Day 0 regardless of skill).
- **Outcome Goals = measurable results** (metric + target). **Projects = activities.** Don't blur them.
- **Honesty lock (proof-only):** no task may fabricate a metric, case study, store detail, email, or track record. Credibility comes from made-for-them samples + documented client deltas only.
- **Canonical names (no aliases):** Projects **A / B / C** (active), **N1–N2** (Next-Up), **F1–F2** (Future), **SM1–SM5** (Someday-Maybe). Goals **VG-1/VG-2**, **OG-1…OG-6**.
